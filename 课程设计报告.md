# 银行家算法资源分配模拟系统课程设计报告

## 一、课设题目

银行家算法资源分配模拟系统。

本课设选择操作系统中的死锁避免问题作为研究对象，使用 C++ 实现银行家算法，对多进程、多资源类型环境下的资源分配过程进行模拟。系统能够根据进程最大需求、已分配资源和当前可用资源判断系统是否安全，并在进程请求资源时决定是否允许分配。

## 二、需求分析

### 2.1 功能需求

系统需要完成以下功能：

1. 初始化系统资源状态，包括进程数量、资源类型数量、系统总资源、最大需求矩阵 Max 和已分配矩阵 Allocation。
2. 根据 Max 和 Allocation 自动计算 Need 矩阵。
3. 显示系统当前资源状态，包括 Available、Max、Allocation、Need。
4. 执行安全性检查，判断当前系统是否存在安全序列。
5. 模拟进程请求资源，判断请求是否满足银行家算法规则。
6. 对安全请求进行分配，对不安全请求进行拒绝并回滚。
7. 模拟进程释放资源，并更新系统状态。
8. 提供帮助命令，指导用户完成操作。
9. 对错误输入给出明确异常提示，避免程序中断。

### 2.2 交互需求

系统采用终端命令方式进行交互。用户可以通过如下命令操作系统：

| 命令 | 功能 |
| --- | --- |
| `help` | 查看全部命令 |
| `help <command>` | 查看指定命令用法 |
| `demo` | 载入经典银行家算法样例 |
| `new <进程数> <资源类型数>` | 创建并初始化新系统 |
| `show` | 显示资源矩阵 |
| `safe` | 执行安全性检查 |
| `request <pid> <r1> ... <rm>` | 请求资源 |
| `release <pid> <r1> ... <rm>` | 释放资源 |
| `reset` | 清空系统状态 |
| `exit` / `quit` | 退出系统 |

### 2.3 异常处理需求

系统需要对以下错误进行处理：

1. 未知命令。
2. 命令参数数量错误。
3. 参数不是非负整数。
4. 进程编号越界。
5. 请求向量长度与资源类型数量不一致。
6. 请求资源数量大于 Need。
7. 请求资源数量大于 Available。
8. 请求后系统进入不安全状态。
9. 释放资源数量大于 Allocation。
10. 未初始化系统时执行资源操作。

## 三、概要设计

### 3.1 系统结构

系统采用模块化设计，避免将所有代码写在 `main.cpp` 中。

| 模块 | 主要职责 |
| --- | --- |
| `main.cpp` | 程序入口，只负责设置控制台环境并启动界面 |
| `Encoding.h/.cpp` | 处理 Windows 终端中文编码，减少运行窗口乱码 |
| `Types.h` | 定义公共数据结构、命令类型和结果类型 |
| `BankerSystem.h/.cpp` | 实现银行家算法核心逻辑 |
| `CommandParser.h/.cpp` | 解析终端命令并检查基本语法 |
| `ConsoleUI.h/.cpp` | 负责终端交互、提示信息和初始化引导 |

### 3.2 数据结构

系统使用以下资源矩阵和向量：

| 名称 | 含义 |
| --- | --- |
| `Available` | 当前系统可用资源向量 |
| `Max` | 每个进程对各类资源的最大需求矩阵 |
| `Allocation` | 当前已经分配给每个进程的资源矩阵 |
| `Need` | 每个进程还需要的资源矩阵，`Need = Max - Allocation` |

## 四、详细设计

### 4.1 安全性检查算法

安全性检查用于判断当前系统是否处于安全状态。

算法步骤如下：

1. 设置工作向量 `Work = Available`。
2. 设置完成标记数组 `Finish[i] = false`。
3. 在所有未完成进程中寻找满足 `Need[i] <= Work` 的进程。
4. 若找到进程 `Pi`，则认为该进程可以顺利执行完成，并释放其资源：
   `Work = Work + Allocation[i]`。
5. 将 `Finish[i]` 置为 `true`，并将 `Pi` 加入安全序列。
6. 重复上述过程，直到没有新的进程可以加入安全序列。
7. 如果所有进程的 `Finish` 都为 `true`，则系统安全；否则系统不安全。

### 4.2 资源请求算法

当进程 `Pi` 请求资源 `Request` 时，系统按以下步骤处理：

1. 判断 `Request <= Need[i]`，否则说明进程请求超过最大声明需求，拒绝分配。
2. 判断 `Request <= Available`，否则说明当前可用资源不足，拒绝分配。
3. 试探性分配资源：
   - `Available = Available - Request`
   - `Allocation[i] = Allocation[i] + Request`
   - `Need[i] = Need[i] - Request`
4. 执行安全性检查。
5. 如果系统仍安全，则正式分配资源。
6. 如果系统不安全，则撤销试探性分配，恢复原状态并拒绝请求。

### 4.3 资源释放算法

当进程 `Pi` 释放资源 `Release` 时，系统按以下步骤处理：

1. 判断 `Release <= Allocation[i]`。
2. 若释放量超过已分配量，拒绝释放。
3. 更新系统状态：
   - `Allocation[i] = Allocation[i] - Release`
   - `Available = Available + Release`
   - `Need[i] = Need[i] + Release`

### 4.4 输入校验设计

系统通过 `CommandParser` 和 `ConsoleUI` 共同完成输入校验：

1. 命令解析阶段检查命令名、参数数量和非负整数格式。
2. 交互初始化阶段检查向量长度、矩阵维度和资源数范围。
3. 算法执行阶段检查进程编号、请求合法性和释放合法性。
4. 所有错误都会输出提示信息，用户可以根据提示重新输入。

## 五、实现说明

### 5.1 模块职责

`BankerSystem` 是系统核心类，负责保存资源状态，并提供初始化、安全性检查、资源请求和资源释放等接口。

`CommandParser` 用于将用户输入的一行字符串转换为命令对象，避免在界面层直接处理复杂字符串。

`ConsoleUI` 是用户交互层，负责显示菜单、读取输入、调用核心算法并输出结果。

`Encoding` 用于在 Windows 环境下将程序内部 UTF-8 文本转换为系统本地代码页输出，避免 CLion 或传统控制台将中文显示为乱码。

`main.cpp` 只负责程序启动，符合模块化设计要求。

### 5.2 健壮性设计

为防止错误输入导致程序崩溃，系统进行了多层保护：

1. 所有资源值必须为非负整数。
2. 进程数量限制为 1 到 20。
3. 资源类型数量限制为 1 到 10。
4. 单个资源值最大为 100000，避免异常大数。
5. 请求资源前必须先初始化系统。
6. 非法请求不会改变系统状态。
7. 释放资源时不能超过进程已持有资源。

## 七、总结

本系统使用 C++ 实现了银行家算法资源分配模拟，能够展示操作系统死锁避免的基本思想。程序采用多文件模块化结构，使算法逻辑、命令解析和终端交互相互分离，提升了代码可读性和可维护性。

系统不仅能够完成正常的资源请求和安全性判断，还能对多种错误输入给出明确提示，避免因用户错误输入导致程序异常终止，满足课程设计中关于可用性、异常提示和健壮性的要求。
